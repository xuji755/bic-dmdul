# 佰晟智算发布开源工具 bic-dmdul：面向达梦数据库的离线数据灾难恢复工具

在数据库运维和故障处置场景中，最棘手的问题之一是：数据库实例已经无法启动，但数据文件仍然存在，业务又急需尽可能恢复关键表数据。

针对这一类故障场景，佰晟智算（深圳）技术有限公司正式发布开源工具 `bic-dmdul`。这是一个面向达梦 DM8 数据库的离线数据抽取与灾难恢复工具，定位类似 Oracle 生态中的 DUL 工具：当数据库无法正常打开时，直接从底层数据文件、系统字典和存储结构中恢复表数据。

## 为什么需要 bic-dmdul

数据库无法启动的原因很多，例如控制文件异常、系统表空间损坏、字典信息不可用、误操作、存储故障、实例异常宕机后无法恢复等。在这些场景下，常规 SQL 查询、导出工具、备份恢复链路可能都无法使用。

如果数据文件仍然保存完好，仍然可能存在恢复机会。关键问题是：如何在不依赖数据库实例启动的情况下，识别数据文件、表空间、对象字典、段、页、行和字段，并把业务表数据重新导出为可重装载的数据格式。

`bic-dmdul` 的目标正是解决这个问题。它尝试绕过数据库实例执行层，直接解析达梦数据库底层存储结构，从离线文件中恢复用户需要的表数据。

## bic-dmdul 的核心目标

`bic-dmdul` 的核心目标可以概括为一句话：

当达梦数据库无法启动，但数据文件仍可读取时，尽可能从离线文件中恢复用户表数据，并生成可审计、可重装载的导出结果。

围绕这个目标，工具重点解决以下问题：

- 识别离线数据库目录中的控制文件和数据文件；
- 从控制文件和文件头中恢复表空间、数据文件、group、file 编号等基础信息；
- 从 `SYSTEM.DBF` 中解析 SYS 系统字典表，生成离线字典；
- 根据离线字典定位用户表、列定义、storage id、root page 和分区入口；
- 解析 BTREE 表页、行结构、字段类型和 LOB 数据；
- 将恢复出的数据导出为 DUL 文本、raw-safe row archive 或带 LOB 附件的结构化结果；
- 提供导入工具，把导出结果转换成可在目标达梦数据库中执行的重建 SQL。

## 当前主要功能

### 1. 离线字典 Bootstrap

`bic-dmdul` 可以从达梦数据文件中构建离线字典，包括：

- `file.dict`：数据文件、表空间和 group 映射；
- `user.dict`：用户信息；
- `tab.dict`：表对象、分区对象、storage 入口；
- `col.dict`：列名、类型、长度、精度等字段定义；
- `storage_scan.dict`：在系统字典缺失时，通过扫描数据文件生成的候选 storage 信息。

这些字典文件是后续离线导出的基础。完成 bootstrap 后，后续导出命令可以直接使用 dict 文件，避免反复扫描系统字典。

### 2. 表数据离线导出

工具可以根据离线字典定位表的 storage root，遍历 BTREE 页并解码行数据。当前支持多种导出格式：

- DUL 文本格式，便于查看和重装载；
- 二进制 row archive，保留物理行 bytes 和建表脚本；
- parts manifest，用于多线程导出和分片导入；
- LOB 外置附件或 row archive 内嵌 LOB block。

对于大表、分区表和多表导出，工具支持并发导出能力。

### 3. 导入与重建

`bic-dmdul` 不只负责导出，还提供导入侧工具。导入工具可以读取 DUL、row archive 或 parts manifest，并生成可在目标达梦数据库执行的 SQL。

导出文件头部会包含建表脚本，导入时可以用于重建目标表结构。对于 row archive 格式，工具保留了更强的 raw-safe 能力，能够减少分隔符、换行、二进制字符对重装载过程的影响。

### 4. LOB 数据恢复

工具已经支持已验证的短内联 LOB 和 out-of-line LOB 页链读取。对于 BLOB、CLOB、TEXT 等大字段，工具可以把 LOB 数据作为附件或 row archive 内嵌 block 保存。

如果遇到无法解析的 LOB locator，工具不会伪造成功结果，而是保留 locator 证据并报告诊断信息，方便后续人工分析或继续研发。

### 5. 分区表支持

对于分区表，`bic-dmdul` 会识别叶子分区 storage，并默认导出所有叶子分区。也可以指定导出某一个分区或多个分区列表。

这对于大型分区表非常重要，因为很多生产系统的核心业务表都采用分区设计。工具在设计上避免只导出父对象而遗漏实际分区数据。

### 6. TRUNCATE 和 DROP 后恢复探索

在某些误操作场景中，表被 `TRUNCATE` 或 `DROP` 后，旧数据页可能尚未被覆盖。`bic-dmdul` 提供 storage 级恢复能力：

- 对 `TRUNCATE` 场景，可以基于旧 storage id 扫描残留数据页；
- 对 `DROP` 场景，可以扫描当前字典中未引用的 orphan storage；
- 如果字段列表仍可获得，可以按原字段结构导出；
- 如果字段列表不可获得，可以先导出 raw 行并给出字段候选推断。

这类恢复属于灾难恢复模式，不等价于数据库一致性读，因此工具会尽量保留页号、行偏移、raw bytes 和诊断信息，避免把不确定恢复包装成确定成功。

### 7. 存储过程和索引 DDL 导出

除表数据外，`bic-dmdul` 还支持按需导出用户存储过程源码，并生成创建脚本。工具也支持生成普通索引的创建脚本，用于恢复后重建对象结构。

当前复杂索引类型，例如函数索引、位图索引、虚拟索引等，仍作为后续增强方向。

### 8. 压缩 HUGE 表研究与严格模式

达梦 `HUGE TABLE` 是列存储结构，与普通 BTREE 行存表不同。当前工具已经验证一种压缩 `HUGE TABLE ... COMPRESS LEVEL 1 FOR 'QUERY LOW'` 的恢复路径：通过内部 `$RAUX` BTREE storage 恢复主表逻辑行，并完成导出、导入和双向比对。

同时，研发过程中也确认了 `QUERY HIGH` 形态会产生 `$AUX.CPR_FLAG='Y'` 的列压缩区，真实数据位于 `HMAIN/SCH<schema_id>/TAB<object_id>/COL*.dta` 这类列文件中。对于这类列压缩区，工具已经定位到物理文件和 zlib 变长列结构，但尚未实现完整通用的列区解压、定长列编码解析和按行重组。

因此，`bic-dmdul` 在严格模式下会把不完整恢复风险显式暴露。例如 `$RAUX` 代理映射会报告 `huge-raux-proxy-mapping`，`dump-data --strict` 会通过 `strict_ok=false`、`tables_strict_failed>0` 和非零退出码提醒用户：当前结果不能直接宣称为完整恢复。

## 设计原则：保真优先，证据优先

灾难恢复工具最怕“看起来成功”。如果导出命令执行完成，但数据不完整、字段错位、LOB 丢失或分区遗漏，这种成功反而会误导用户。

因此，`bic-dmdul` 的设计原则是：

- 能确定恢复的，完整导出并支持导入比对；
- 不能确定的，保留 raw evidence；
- 有风险的，报告 diagnostics；
- 严格模式下，不完整或不确定恢复必须失败；
- 不依赖在线 `DBA_*` 视图作为最终恢复路径；
- 每一项重要能力都要通过“导出、导入、比对”的完整链路验证。

这种设计让工具不仅能“抢救数据”，也能让恢复过程可解释、可审计、可复盘。
